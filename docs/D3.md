# Documento D3 - ReValue

## 1. Introduzione

ReValue e una piattaforma per favorire il riuso di oggetti tra utenti. Il donatore pubblica un annuncio con descrizione, categoria, scadenza e posizione; l'acquirente consulta il catalogo, prenota l'oggetto e completa lo scambio tramite QR Code. Il sistema assegna punti wallet a scambio concluso e mantiene controlli su prenotazioni, ritiri, messaggi e segnalazioni.

### 1.1 Recap funzionale

- Catalogo pubblico con filtri per categoria, dimensione, materiale, scadenza e distanza.
- Registrazione e login con JWT.
- Gestione annunci con creazione, modifica, cancellazione logica e stati.
- Prenotazioni con protezione da doppia prenotazione concorrente.
- Scambio tramite QR Code generato dal donatore e validato dall'acquirente.
- Wallet con saldo e storico transazioni.
- Chat/messaggi tra partecipanti allo scambio.
- Funzioni admin per segnalazioni, utenti e annunci.

### 2.2 Flusso backend

```text
Client web/mobile
  -> Express app
  -> middleware globali (CORS, body parser, morgan)
  -> routes /api/*
  -> middleware di sicurezza (authenticate, notSospeso, requireAdmin)
  -> controller
  -> service quando presente
  -> model Mongoose
  -> MongoDB
```

I controller restano responsabili delle risposte HTTP. La logica auth e stata estratta in `backend/src/services/authService.js`; wallet e chat hanno gia service dedicati. Gli errori non intercettati e le route API inesistenti passano da `backend/src/middleware/errorHandler.js`, che restituisce il formato:

```json
{
  "ok": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "Route GET /api/example non trovata"
  }
}
```

## 6. Note Finali

- Il logging HTTP e attivo con `morgan('combined')` in produzione e `morgan('dev')` in sviluppo.
- Le route API non trovate restituiscono un errore standardizzato.
- La logica auth e isolata nel service layer per rendere i controller piu sottili e testabili.
- La validazione JWT gestisce esplicitamente token scaduti, token invalidi e segreto mancante.
